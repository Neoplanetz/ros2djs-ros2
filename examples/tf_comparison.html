TFClient Comparison — v1.3

Both canvases subscribe to the same /pose topic. The left one ignores header.frame_id (v1.2 behavior). The right one composes the pose with map ← frame_id via TFClient (v1.3 new path).

Setup (ROS 2, 3 terminals)

  1. rosbridge 
    ros2 launch rosbridge_server rosbridge_websocket_launch.xml
  2. static transform — test_frame lives at (3, 2) in map: 
    ros2 run tf2_ros static_transform_publisher --x 3 --y 2 --z 0 --roll 0 --pitch 0 --yaw 0 --frame-id map --child-frame-id test_frame
  3. publish a pose at the origin of test_frame: 
    ros2 topic pub /pose geometry_msgs/msg/PoseStamped "{header: {frame_id: 'test_frame'}, pose: {position: {x: 0.0, y: 0.0, z: 0.0}, orientation: {w: 1.0}}}" -r 10

Expected: left arrow sits at (0, 0) — frame ignored. Right arrow sits at (3, 2) — TF applied.

Try this next: change --x 3 --y 2 to other values in terminal 2 (restart the command). The right canvas follows; the left stays put. Or switch frame_id to map in terminal 3 — both sides agree (identity transform).

Without tfClient

v1.2 behavior: frame_id ignored; pose drawn verbatim.

With tfClient

v1.3: pose composed with TF transform into fixedFrame: 'map'.

Multi-robot scenario

For the multi-robot case that motivated this feature, replace terminal 2 with two static publishers and publish to two topics (or a single /markers MarkerArray mixing robot_0/odom and robot_1/odom frame_ids). Without tfClient they collapse to one spot; with tfClient they separate.

ros2 run tf2_ros static_transform_publisher --x  5 --y 0 --z 0 --roll 0 --pitch 0 --yaw 0 --frame-id map --child-frame-id robot_0/odom
ros2 run tf2_ros static_transform_publisher --x -5 --y 0 --z 0 --roll 0 --pitch 0 --yaw 0 --frame-id map --child-frame-id robot_1/odom
